Software Development Life Cycle (SDLC)

The Software Development Life Cycle (SDLC) is a structured process used for developing software applications. Below is how we typically deliver software here at Sainsbury's Tech. For each stage of the life cycle, I've listed the definition and what happens (aka what my team try to follow), inputs, outputs, and who is typically involved.

Planning üß≠

This is where you figure out what the project is all about - what you're trying to achieve, what problems you're solving, and roughly how you'll get there. It's also where you think about who needs to be involved, what tools or resources you'll need, any deadlines, and any risks you might face along the way.

  • Inputs: Business goals, early user feedback, high-level ideas, market/competitor insight.
  • Outputs: Roadmap, initial scope and risks, high-level timeline, rough sizing/ballpark estimates (guesstimates).
  • Who is involved: Product Manager (PM), Delivery/Scrum Master (we don't do this, but, my team assign sprint captains), Engineers (for insight, how doable is this?), UX (User Experience) Designer, BA (Business Analyst, although my team currently doesn't have one as of yet), key stakeholders.

Requirements

Here, you pin down exactly what the system or product needs to do. This could be features, behaviours, or even rules the system has to follow. Basically, you're turning the ‚Äúwhat‚Äù from the planning stage into a clear to-do list that everyone agrees on.

  • Inputs: Planning outcomes, stakeholder meetings with EM (Engineering Manager) and PM, analytics, existing system constraints.

  • Outputs: Prioritised backlog (stories & acceptance criterias), definition of ready, updated release plan, refined risks.

  • Who is involved: PM, EM, BA, UX Designer, Engineers, QA (Quality Assurance) Testers, Architecture/Security where needed.

Design üé®

This stage is creating the blueprint before building something. You decide how everything will fit together - from the layout and user experience to the technical details like databases and APIs.

  • Inputs: User stories & acceptance criterias, design system, technical standards, spikes/prototypes.

  • Outputs: UX artefacts (flows/wireframes), API specs, architecture notes, definition of done, test strategy outline.

  • Who is involved: UX/UI Designers, Engineers, Senior Engineers, QA, PM/BA for validation.

Development

This is the actual building stage, where you write the code and bring the design to life. It's usually done in smaller chunks (aka sprints) so you can test and adjust as you go along.

  • Inputs: Design specifications, wireframes, prototypes, architecture diagrams, feature flags.

  • Outputs: Working in increments, passing unit tests, code reviews and merged PRs, documentation updates.

  • Who is involved: Engineers (all levels), QA Testers, EM/PM/BA for oversight.

Testing ‚úÖ

Before anything goes live, you want to check everything works as intended. This means looking for bugs, making sure everything runs smoothly, and confirming that it meets the requirements set in earlier stages.

  • Inputs: Completed code, test cases, user stories, acceptance criteria.

  • Outputs: Test results, bug reports, feedback for developers, sign-off for deployment (acceptance).

  • Who is involved: QA Testers, Engineers (for fixing bugs), EM/PM/BA for validation.

Deployment

This is when you release the work so people can actually use it. Sometimes it's done in stages, and you'll often keep a close eye on things straight after in case any problems pop up.

  • Inputs: Approved code, test results, deployment plan, rollback procedures (just in case).

  • Outputs: Live system, deployment logs, monitoring setup, user documentation.

  • Who is involved: Engineers, QA Testers, EM/PM for communication.

Maintenance

Once it's live, you still need to look after it. That could mean fixing bugs, updating it with new features, or making improvements to keep it running well.

  • Inputs: Monitoring/alerting, customer feedback, incident reports, security alerts, analytics.

  • Outputs: Hotfixes/patches, improve system performance, update documentation, learning for future sprints.

  • Who is involved: Engineers (on-call if and when needed), QA Testers, EM/PM for support.
